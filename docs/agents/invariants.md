# Critical Invariants

If you violate these, the system breaks. Each row cites the enforcement location ā€”
when in doubt, read the cited module's `//!` doc and the named symbol (symbols
outlive line numbers).

| # | Invariant | Enforcement |
|---|-----------|-------------|
| 1 | **Migrations are append-only.** Never edit an applied `migrations/NNNN_*.sql` ā€” add a new numbered file. `sqlx::migrate!("./migrations")` tracks applied versions in `_sqlx_migrations`. Forward-only: there is no down migration. | `src/repo/db.rs` ā€” `connect()`; pinned by the `migration_rerun_is_idempotent` test |
| 2 | **WAL + `foreign_keys=ON` on every pool.** WAL allows concurrent reads during writes. Every FK is `ON DELETE RESTRICT`; without the pragma SQLite silently ignores FK constraints. | `src/repo/db.rs` ā€” `connect()` pragmas; asserted by `connect_sets_wal_and_foreign_keys` |
| 3 | **Worktrees are ephemeral.** Never edit files under `~/.boi/v2/worktrees/` directly ā€” the spec-level `teardown` step removes the integration worktree and every task worktree. | `src/runtime/worktree.rs` ā€” `default_worktree_root()`, `teardown()` |
| 4 | **Verify commands must be idempotent.** Verify re-runs: the executeā†”review `Redo` loop re-enters the verify path each iteration, and a worker may call the MCP `verify_run` tool any number of times. `CREATE TABLE`-style verify breaks retries. | Convention; re-run paths: `src/service/routing.rs` `route_task`, `src/runtime/tool_host.rs` `run_verification` |
| 5 | **Malformed phase/routing config is a loud startup rejection.** Phase and pipeline TOMLs (`~/.boi/v2/phases/`, `~/.boi/v2/pipelines/`) parse with `deny_unknown_fields`; `validate_pipeline` runs once at `Orchestrator::new`, and a malformed routing graph aborts daemon boot ā€” never a silent mid-run stall. | `src/config/phase.rs` `PhaseDef`; `src/service/routing.rs` `validate_pipeline`; `src/cli/boot.rs` `BootError` |
| 6 | **One daemon; the bus disposes.** The orchestrator + `EventBus` live in the single long-running `boi daemon`. The bus is the sole chokepoint for state transitions: it checks `check_task`/`check_spec` against the persisted state before writing any state row ā€” workers PROPOSE, the bus DISPOSES. | `src/service/bus.rs` (chokepoint), `src/service/transitions.rs` ā€” `check_task`/`check_spec` |
| 7 | **Write-side CLI goes through the control socket.** `dispatch` / `cancel` / `unblock` / `resolve-conflict` / `fail` submit a typed `DaemonCommand` over `~/.boi/v2/daemon.sock`; with no daemon running they fail loud with a non-zero exit ā€” a DB-only status flip with a live orphan worker is forbidden. Read-only commands (`dashboard`, `log`, `traces`, `spec show`) read SQLite/DuckDB directly. | `src/cli/mod.rs` module doc (process model), `src/cli/control.rs` |
| 8 | **Layer deps are forward-only.** `types ā†’ config ā†’ repo ā†’ service ā†’ runtime ā†’ cli`; a module may `use crate::<other>` only when `<other>` is a lower layer (intra-layer is fine). Nothing imports `cli/`. | `scripts/checks/module-dep-audit.sh`, run by `just lint-scripts` (part of `just ci`) |
| 9 | **Every spawned build child gets the shared warm `CARGO_TARGET_DIR`** (`~/.boi/v2/cargo-target`, OBS-032) ā€” both the `goose` worker spawn path and `validate`'s verification commands. Consequence for gate authors: `cargo` artifacts do NOT land in the worktree's `target/`, so a gate like `cargo build --release && test -x target/release/<bin>` fails deterministically (the build goes to the shared dir; the relative check resolves against the worktree). Check artifacts via `${CARGO_TARGET_DIR:-target}/release/<bin>`, and pre-test gates locally with `CARGO_TARGET_DIR` exported to match. Never symlink a worktree `target/` to the shared dir ā€” it holds other specs' uplifted binaries, so an existence check would false-pass (a silent wrong-pass, strictly worse than the loud fail). A daemon-env `CARGO_TARGET_DIR` overrides the default; a shell-level `export` inside the command wins over both. | `src/runtime/worktree.rs` `resolve_cargo_target_dir`; injected at `src/runtime/validate.rs` `run_command` and the `src/runtime/goose.rs` worker spawn; pinned by `test_l2_run_command_child_env_carries_the_shared_cargo_target_dir` |
